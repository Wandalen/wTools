# Integration: former_types

### Scope

- **Purpose**: Document the dependency on `former_types` as the runtime type contract for generated code.
- **Responsibility**: Describe what generated macro output references from `former_types` and why.
- **In Scope**: What generated token streams reference from `former_types` and compatibility requirements.
- **Out of Scope**: The `former_types` crate itself — see that crate's own documentation.

### System Description

`former_types` is the runtime type crate in the former ecosystem. It defines the traits
and types that generated former infrastructure implements and uses. Because this crate
is a proc-macro crate, it cannot itself export runtime types; all runtime contracts are
delegated to `former_types`.

### Integration Points

The token streams emitted by `Former` macro expansion reference types and traits from
`former_types`. The generated former struct implements traits defined there. The
generated storage struct stores fields in forms that satisfy those type contracts.

`former_types` is listed as a dependency in `Cargo.toml` with the `types_former` feature
active, gated behind the `enabled` feature.

### Error Handling

Type mismatches between generated code and `former_types` trait definitions surface as
compile errors in consumer crates. These errors point to the expanded macro output, not
to this crate directly.

### Compatibility Requirements

Breaking changes to traits in `former_types` require corresponding updates to the token
streams generated by this crate. Any `former_types` version bump that changes trait
signatures must be validated against all generated former patterns.
