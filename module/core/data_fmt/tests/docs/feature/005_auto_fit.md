# Feature: Auto-Fit

### Scope

- **Purpose**: Drive test coverage for the auto-fit feature.
- **Responsibility**: Documents test cases for the combined auto-wrap and auto-fold auto-fit pipeline in `docs/feature/005_auto_fit.md`.
- **In Scope**: Natural fit (no auto-fit), flex column wrapping, column folding with Labeled style, `auto_wrap(false)` opt-out, `terminal_width` override, CSV/TSV bypass, `$COLUMNS` environment variable tier (Tier 1), tty-query tier (Tier 2) coverage, 120-column fallback (Tier 3).
- **Out of Scope**: Budget allocation algorithm details (see `algorithm/004`); fold detection algorithm details (see `algorithm/005`).

### FT-1: table that fits naturally renders without wrapping or folding

- **Given:** A table whose total content width (plus separators) is less than or equal to `terminal_width`.
- **When:** The table is rendered with default `TableConfig` (auto_wrap and auto_fold both true).
- **Then:** Output matches the pre-auto-fit rendering; no cells are wrapped; no continuation lines appear; column widths are content-driven.

---

### FT-2: flex column exceeding budget wraps to additional sub-lines

- **Given:** A table with a wide column that causes total width to exceed `terminal_width(80)`.
- **When:** The table is rendered with `auto_wrap(true)` (default).
- **Then:** The wide column's cell content wraps across multiple sub-lines within its budget; row height grows to accommodate the wrapped content; no content is lost (truncated).

---

### FT-3: overflow columns fold to labeled continuation lines

- **Given:** A table where even after wrapping, total row width exceeds `terminal_width`.
- **When:** The table is rendered with `auto_fold(true)` (default) and `fold_style(Labeled)`.
- **Then:** Overflow columns are moved to labeled continuation lines below the row (format: `ColumnName: value`); folded values are indented by `fold_indent`; the primary row fits within `terminal_width`.

---

### FT-4: auto_wrap(false) disables cell wrapping strategy

- **Given:** A table where a flex column exceeds budget; `auto_wrap(false)` is set.
- **When:** The table is rendered.
- **Then:** Cell content is not wrapped; rows remain single-line regardless of budget overflow; no sub-lines are generated by the auto-fit pipeline.

---

### FT-5: terminal_width override takes priority over auto-detection

- **Given:** A `TableConfig` with `terminal_width(Some(60))` and a table that would fit in 80 but not 60 columns.
- **When:** The table is rendered (even when run in a wider terminal or no TTY).
- **Then:** The formatter uses 60 as the effective width; auto-fit activates as if the terminal is 60 columns wide; output fits within 60 characters per line.
- **Note:** FT-5 and FT-7 both test `terminal_width(Some(60))` with wrapping as the
  observable effect. FT-5 establishes the priority principle (override beats detection);
  FT-7 is a concrete measurement test (line count ≤ 60). Both are useful: FT-5 tests
  the tier ordering contract; FT-7 tests the output constraint.

---

### FT-6: CSV and TSV presets disable auto-fit automatically

- **Given:** A table with long cell values rendered via `TableConfig::csv()` or `TableConfig::tsv()`.
- **When:** The table is rendered.
- **Then:** No cell wrapping occurs; no column folding occurs; the CSV/TSV output is single-line-per-row regardless of content width.

---

### FT-7: explicit terminal_width(60) constrains output to 60 columns

- **Given:** A `TableConfig` with `terminal_width(Some(60))` (Tier 0 explicit
  override); the table content is wider than 60 but narrower than 120 columns.
- **When:** The table is rendered.
- **Then:** The formatter uses 60 as the effective terminal width; content that
  exceeds 60 columns triggers auto-fit; all output lines fit within 60 characters.
- **Note:** The Tier 0 explicit `terminal_width(Some(60))` override takes priority over
  all other tiers including the `COLUMNS` env var (Tier 1) and the tty-query (Tier 2).

---

### FT-8: tty-query tier traversed before 120-column fallback

- **Given:** No `terminal_width` override in `TableConfig`; running in a non-TTY
  environment (e.g. `cargo nextest`); the table content is narrower than 120 columns.
- **When:** The table is rendered.
- **Then:** `resolve_terminal_width()` traverses Tier 1 (`$COLUMNS` env var),
  which returns `None` when `COLUMNS` is unset; Tier 2 (tty query via
  `terminal_size` crate) then queries the TTY, which also returns `None` in
  non-TTY environments; Tier 3 (120-column hardcoded fallback) activates; the
  narrow table renders fully without wrapping or folding; all output lines are
  within 120 columns.
- **Note:** Resolution order is: Tier 0 (explicit `Some(w)` config override) →
  Tier 1 (`$COLUMNS` env var) → Tier 2 (`terminal_size` tty query) → Tier 3
  (120 fallback). In non-TTY test environments Tier 2 returns `None`, so Tier 3
  activates and the narrow table renders without wrapping or folding.

---

### FT-9: COLUMNS env var controls auto-fit terminal width in non-TTY environments

- **Given:** `COLUMNS=40` is set in the process environment; a table whose natural total
  row width is ~90 characters; no explicit `terminal_width` override in `TableConfig`.
- **When:** The table is rendered with `auto_wrap(true)` (default).
- **Then:** `resolve_terminal_width()` reads `COLUMNS`, parses `40` as a positive `u16`,
  and returns `40` as the effective width; the auto-fit pipeline treats the terminal as
  40 columns wide; all output lines fit within 40 characters; setting `COLUMNS` to an
  empty string, a non-numeric value, or `0` causes the env var to be ignored and
  resolution falls through to Tier 2 (tty query) and then Tier 3 (120 fallback).
- **Note:** This test must serialize access to the `COLUMNS` env var (via `Mutex` or
  `#[serial]`) to prevent interference with parallel test threads. Invalid `COLUMNS`
  values (empty string, non-numeric, `"0"`) must each fall through to the next tier
  — this behavior is part of the same test case assertion, not a separate case.
