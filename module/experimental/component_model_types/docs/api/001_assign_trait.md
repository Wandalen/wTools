# API: Assign Trait System

### Scope

- **Purpose**: Document the public trait interface for type-safe component assignment used across the wTools ecosystem.
- **Responsibility**: Provides the canonical API reference for callers implementing or consuming the Assign trait system.
- **In Scope**: The Assign, OptionExt, AssignWithType, and PopularType traits — their methods, semantics, and blanket implementations.
- **Out of Scope**: Derive macro implementations (→ `component_model_meta`); standard library type impls (→ generated by macro, see invariant/001).

### Abstract

A four-trait API surface enabling generic, type-safe component assignment. The central trait accepts any value convertible to the target component type, making assignment flexible without sacrificing type safety. A consuming variant enables builder-style chaining. An Option-aware extension and a blanket explicit-type variant round out the surface.

### Operations

**Assign trait** — generic assignment interface parameterized by component type and input type:

| Method | Semantics | Returns |
|--------|-----------|---------|
| `assign` | Mutating assignment — sets the component on the receiver | nothing |
| `impute` | Consuming assignment — sets the component, returns the modified receiver | receiver (for chaining) |

The two type parameters distinguish which component is being set (`T`) from what value is being accepted (`IntoT`). This enables a single receiver type to implement assignment for multiple distinct components without ambiguity.

**OptionExt trait** — assignment extension for optional fields:

| Method | Semantics |
|--------|-----------|
| `option_assign` | If the Option is `None`, creates a new value; if `Some`, updates the existing value |

Sealed — only implemented for `Option<T>` where the inner type itself implements Assign for itself.

**AssignWithType trait** — blanket implementation over all Assign implementors:

| Method | Semantics |
|--------|-----------|
| `assign_with_type` | Explicit type-guided assignment; disambiguates when multiple Assign implementations exist for the same receiver |

**PopularType marker** — identifies standard library types (Duration, PathBuf, etc.) that receive special treatment in the derive macros. Carries no methods; implementation signals to the macro layer that expanded assignment impls should be generated.

### Error Handling

All assignment operations are infallible by design. There is no validation at assignment time — values are accepted if they satisfy the type-level Into constraint. Validation is the caller's responsibility.

### Compatibility Guarantees

- Adding methods to Assign is a breaking change for all external implementors.
- The sealed OptionExt trait can have methods added without breaking external code (sealed = no external implementors).
- AssignWithType is a blanket impl — its behavior tracks Assign; no direct implementor burden.
- PopularType is a marker; adding or removing standard library implementations is a breaking change for downstream code relying on derive-generated impls.
