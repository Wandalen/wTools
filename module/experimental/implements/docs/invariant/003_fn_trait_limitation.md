# Invariant: Fn Trait Limitation

### Scope

- **Purpose**: Warn callers that applying the macro to a named function item against a callable trait bound produces a compile-time error, not a false result.
- **Responsibility**: Documents the Fn trait limitation — its precise statement, structural cause, and the workaround that avoids the compile error.
- **In Scope**: The compile-time behavior of the macro when the inspected value is a named function item and the bound is a callable trait.
- **Out of Scope**: Checking callable traits on closures or function pointers (which works correctly), all non-callable trait bounds (which work correctly for all value kinds).

### Invariant Statement

For all named function items: applying the macro with a callable trait bound produces a compile-time error. The error is not a false result — the macro expansion fails to typecheck. This holds for all three callable traits. Closures and function pointers typed as callable traits are not subject to this limitation and produce the correct boolean result.

### Enforcement Mechanism

Named function items have unique anonymous types generated by the compiler. These types do not directly satisfy callable trait bounds in the macro's generic constraint context without an explicit coercion. The conditional method implementation cannot be instantiated for a named function item, and the unconditional fallback cannot be reached because type resolution fails before method dispatch. The result is a hard compile error at the macro call site.

### Violation Consequences

Callers who pass a named function directly to the macro with a callable trait bound receive a compile error with no clear indication of the workaround. The fix is to assign the function to a closure binding or an explicitly typed function pointer before passing it to the macro.
